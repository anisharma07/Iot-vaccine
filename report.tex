% ─── Abstract ───────────────────────────────────────────────────────────────
\begin{center}
    \Large\textbf{Abstract}
\end{center}
\vspace{0.5cm}

\noindent
Vaccine cold-chain failures account for up to 50\% of global vaccine wastage, primarily due to inadequate real-time monitoring during transport. This paper presents the design, simulation, and cloud integration of a Smart Vaccine Delivery System built on the ESP32 microcontroller. The system integrates a DHT22 temperature-humidity sensor, a simulated MPU6050 inertial measurement unit, an MQ-135 gas sensor, and a tamper detection switch to provide continuous multi-modal monitoring of vaccine containers in transit. A novel composite \textit{Damage Score Algorithm} aggregates readings from all sensors into a single risk metric (0--10), automatically triggering a CRITICAL alert (status code 9) when the score reaches or exceeds five. All sensor data are transmitted to the ThingSpeak cloud platform every 15 seconds via HTTP over Wi-Fi, enabling remote visualization and historical analysis across eight data fields. The firmware was developed and validated using the Wokwi ESP32 simulator, and a companion real-time web dashboard was built to visualize live telemetry with interactive event simulation. Results confirm reliable detection of all six monitored hazards: cold-chain temperature excursions, tamper events, mechanical shock, unsafe driving, gas leakage, and high humidity. The proposed system offers a cost-effective (approximately INR~1,200--1,400 per unit) and scalable solution for IoT-enabled vaccine cold-chain integrity management.

\vspace{0.6cm}
\noindent\textbf{Keywords:} IoT, ESP32, Vaccine Cold-Chain, DHT22, MPU6050, ThingSpeak, Damage Score Algorithm, Real-Time Monitoring

\section{Introduction}

\subsection{Background and Motivation}

Vaccines are among the most temperature-sensitive pharmaceutical products. The World Health Organization~(WHO) mandates strict cold-chain maintenance at 2\,$^\circ$C--8\,$^\circ$C from the point of manufacture to the point of administration \cite{who_coldchain}. Despite these guidelines, the WHO~estimates that up to 50\% of vaccines are wasted globally each year due to cold-chain failures \cite{who_wastage}. These failures are compounded by: (i)~manual and infrequent monitoring that fails to detect real-time anomalies; (ii)~no sensing of physical damage such as drops, shocks, or excessive vehicle motion; and (iii)~absence of contamination or spoilage detection during transit.

\subsection{Objectives}

The objectives of this work are:
\begin{enumerate}[label=(\roman*)]
  \item Design a multi-sensor IoT node capable of monitoring temperature, humidity, mechanical shock, gyroscopic speed, gas levels, and tamper state.
  \item Develop a composite Damage Score Algorithm that quantifies the cumulative risk to vaccine integrity.
  \item Simulate and validate the system using the Wokwi ESP32 simulator with realistic event scenarios.
  \item Integrate cloud logging via ThingSpeak for remote visibility and historical analysis.
  \item Provide a real-time web dashboard for operational monitoring.
\end{enumerate}

\subsection{Literature Review}
\label{sec:related}

Several recent works have addressed IoT-based vaccine cold-chain monitoring, but each exhibits specific limitations that this work addresses.

Jiang et al.\ \cite{jiang2024} proposed a sustainable IoT framework targeting energy optimization in edge monitoring nodes for cold-chain applications. Their system achieved a 22\% reduction in energy consumption and 99.8\% thermal compliance in real-time cold-chain tests. However, the approach incurs high initial setup costs and struggles in low-connectivity rural regions typical of last-mile vaccine delivery scenarios.

Sujiya et al.\ \cite{sujiya2025} developed a multi-sensor node architecture for continuous transit monitoring with an alert latency under 2~seconds and 98\% fault detection accuracy. Their system monitors only temperature and humidity, omitting physical shock, gas leakage, and multi-fault escalation logic—gaps that this work directly addresses.

Hasanat et al.\ \cite{hasanat2024} designed a smart active vaccine carrier integrating long-range communications for last-mile delivery, achieving 48-hour active uptime and 99\% communication success over a GPS and temperature-based testbed. The approach proposes complex carrier hardware and lacks predictive thermal failure modeling or composite risk scoring.

Enriko et al.\ \cite{enriko2023} presented an integrated end-to-end tracking and cloud logging framework for COVID-19 vaccine distribution, demonstrating reliable visibility across a logistics chain. The system is heavily dependent on continuous network connectivity and provides limited edge-based analytics capability.

Ali and Khan \cite{ali2023} evaluated IoT platforms for COVID-19 vaccine logistics using multi-criteria decision-making models. While the work provides useful platform assessment guidance, it does not contribute a real implementation or sensor fusion strategy.

\smallskip
\noindent\textbf{Gap Analysis:} No prior work combines (i) five sensor modalities, (ii) a composite damage scoring algorithm with severity escalation, (iii) status-code-based event classification, and (iv) a complete cloud-plus-dashboard pipeline within a single low-cost, simulation-validated prototype. This work bridges that gap.

% ─── 2. Problem Statement ────────────────────────────────────────────────────
\section{Problem Statement}
\label{sec:problem}

Existing vaccine transport solutions lack a unified, real-time, multi-parameter monitoring framework that simultaneously covers thermal integrity, mechanical stress, chemical contamination, and physical security. Specifically, three interconnected deficiencies characterize the current state of practice.

\textbf{1. Inadequate thermal monitoring.} Manual and infrequent temperature checks fail to detect transient cold-chain excursions. A vaccine container may breach the 2--8\,$^\circ$C window briefly during a vehicle door opening or refrigerator malfunction, yet pass a spot check taken minutes later—resulting in undetected compromise.

\textbf{2. No physical-damage sensing.} Rough handling, accidental drops, and unsafe driving behaviour can physically damage or contaminate vaccine vials. None of the reviewed prior systems integrate inertial or gyroscopic monitoring to detect shock and vibration events during transit.

\textbf{3. Absence of contamination and security detection.} Gas leakage from co-transported chemicals and unauthorized container opening (tampering) are undetected by any existing low-cost cold-chain monitor. The absence of such detection leads to delayed identification of failures, undetected damage, and consequent waste or administration of compromised vaccines.

The absence of a unified system leads to delayed detection of failures, undetected damage, and consequent waste or administration of compromised vaccines.

\section{Component Descriptions and Characteristics}
\label{sec:components}

\subsection{ESP32 DevKit v4 -- Central Microcontroller}

The ESP32 DevKit v4 is based on the Espressif ESP32-D0WDQ6 dual-core 32-bit Xtensa LX6 processor running at up to 240~MHz \cite{espressif}. It integrates 520~KB of SRAM, 4~MB of flash, and a built-in 802.11 b/g/n Wi-Fi and Bluetooth 4.2 radio, making it ideal for IoT applications that require both local signal processing and cloud connectivity without an external gateway. The board exposes 34 programmable GPIO pins, twelve 12-bit ADC channels, three SPI, two I\textsuperscript{2}C, and two I\textsuperscript{2}S interfaces.

\noindent\textbf{Resistor Selection:} A 220\,$\Omega$ current-limiting resistor was used in series with the red LED. At a 3.3~V GPIO output, this limits LED current to approximately 15~mA, safely below the ESP32's 20~mA per-pin maximum.

\subsection{DHT22 Temperature and Humidity Sensor}

The DHT22 (AM2302) is a calibrated digital sensor with a measurement range of $-40$\,$^\circ$C to $+80$\,$^\circ$C ($\pm$0.5\,$^\circ$C accuracy) and 0\% to 100\%~RH ($\pm$2\%~RH accuracy) \cite{dht22}. It uses a proprietary single-wire digital protocol and is connected to GPIO~4. In this system it enforces the WHO-mandated cold-chain window of 2--8\,$^\circ$C and generates a humidity alert above 80\%~RH to detect condensation risk inside the vaccine container.

\subsection{MPU6050 Inertial Measurement Unit (Simulated)}

The MPU6050 is a six-axis IMU combining a 3-axis accelerometer ($\pm$16\,g) and a 3-axis gyroscope ($\pm$2000\,°/s) in a single package communicating via I\textsuperscript{2}C. In the Wokwi simulation, two potentiometers on GPIO~35 and GPIO~32 substitute for the real device. Their 12-bit ADC output (0--4095) is linearly mapped to [--32768, +32767] to mirror a real MPU6050 16-bit register. The acceleration vector magnitude $|\mathbf{a}| = \sqrt{a_x^2 + a_y^2 + a_z^2}$ (with $a_z = 16384$ as 1\,g baseline) detects shock events, while the gyro channel detects unsafe driving.

\subsection{MQ-135 Gas Sensor (Simulated)}

The MQ-135 is a semiconductor gas sensor sensitive to ammonia, benzene, alcohol, and smoke, with an ADC output range of 0--4095 on a 12-bit channel. It is simulated via a potentiometer on GPIO~34. A raw ADC reading above 400 indicates a gas leakage condition. In a physical deployment the MQ-135 requires a 24-hour burn-in period and a 5~V heater supply; the simulation abstracts this to a direct threshold comparison.

\subsection{Tamper Detection Switch}

A normally-open push button on GPIO~15 (active LOW, internal pull-up enabled) simulates a magnetic reed switch or microswitch mounted on the vaccine container lid. A LOW reading on this pin indicates that the container has been opened, triggering status code~2 (tamper detected) and incrementing the damage score by 2.

\subsection{LCD 16$\times$2 I\textsuperscript{2}C Display Module}

A standard HD44780-compatible 16-character $\times$ 2-row LCD with a PCF8574-based I\textsuperscript{2}C backpack is connected to GPIO~21 (SDA) and GPIO~22 (SCL). It displays the current status code and its text description on row~1 and the damage score on row~2, updating every 2~seconds.

\subsection{Passive Buzzer and Red LED}

A passive piezo buzzer on GPIO~13 emits a 1~kHz tone driven by a \texttt{tone()} call; the red LED on GPIO~2 provides a persistent visual alert. Both are activated simultaneously whenever the damage score reaches or exceeds 5 (CRITICAL, status code~9), providing dual-channel alerting.

\subsection{Sensor Simulation in Wokwi}

Since the MPU6050 IMU and MQ-135 gas sensor are not natively supported as Wokwi components, they are simulated using potentiometers whose analog output (0--4095 ADC counts) is linearly mapped to the expected physical range:

\begin{itemize}
  \item \textbf{Acceleration (GPIO 35, 32):} Potentiometer output is mapped from [0, 4095] to [-32768, 32767], mirroring a real MPU6050 16-bit accelerometer. The vector magnitude $|\mathbf{a}| = \sqrt{a_x^2 + a_y^2 + a_z^2}$ is computed with $a_z$ fixed at 16384 (1\,g baseline).
  \item \textbf{Gas (GPIO 34):} Raw ADC value is used directly, with threshold 400 indicating leakage.
\end{itemize}

The Wokwi simulation circuit includes the ESP32 DevKit C v4, DHT22, three potentiometers, a 16$\times$2 I\textsuperscript{2}C LCD, a push button, a passive buzzer, and a red LED with a 220\,$\Omega$ resistor, as shown in Figure~\ref{fig:wokwi}.

\subsection{Damage Score Algorithm}

The core novelty of this work is the \textit{Damage Score Algorithm}, which aggregates multi-sensor risk indicators into a single composite score in the range [0, 10]. Listing~\ref{alg:damage} defines the scoring logic.

\begin{figure}[H]
\centering
\begin{minipage}{0.88\linewidth}
\begin{lstlisting}[style=cppstyle, caption={Damage Score Algorithm (C++ / Arduino)}, label={alg:damage}]
void evaluateStatus() {
  damageScore = 0;
  status_code = 1;         // default: OK

  if (temp > 8.0 || temp < 2.0) { status_code = 0; damageScore += 3; }
  if (tampered)                  { status_code = 2; damageScore += 2; }
  if (accelMag > 20000)          { status_code = 3; damageScore += 2; }
  if (gyroSpeed > 30000)         { status_code = 4; damageScore += 1; }
  if (gasValue > 400)            { status_code = 5; damageScore += 3; }
  if (humidity > 80.0)           { status_code = 6; damageScore += 1; }

  // CRITICAL override
  if (damageScore >= 5)          { status_code = 9; }
}
\end{lstlisting}
\end{minipage}
\end{figure}

The weighted contributions reflect the relative severity of each failure mode: temperature excursions and gas leaks are most damaging (+3 each), followed by mechanical shock and tamper events (+2 each), and unsafe driving and humidity are lower-severity contributors (+1 each). Any combination reaching a score of five or more elevates the system to CRITICAL status (code 9), triggering continuous buzzer and LED alerts.

\subsection{Discussion}

The system successfully detected all six monitored hazard categories in every simulated scenario. The Damage Score Algorithm proved particularly effective for CRITICAL-level multi-fault escalation: combining gas leakage (+3) with a temperature excursion (+3) immediately triggers CRITICAL status (score $\geq 5$) without requiring either individual fault to reach a higher single-sensor threshold. This approach is more robust than per-sensor alarming, as it accounts for the compounding effect of simultaneous failures on vaccine integrity.

A key strength of the design is its graceful offline capability: the system continues local monitoring, alert actuation, and LCD display even when Wi-Fi is unavailable, logging a warning to the serial monitor and resuming uploads when connectivity is restored.

The primary limitation is that the MPU6050 and MQ-135 are simulated using potentiometers. While the firmware includes commented-out code for real I\textsuperscript{2}C MPU6050 communication and the ADC pin assignments for a real MQ-135, field validation with physical sensors remains future work.

% ─── 7. Future Integration and Large-Scale Applications ─────────────────────
\section{Future Integration and Large-Scale Applications}
\label{sec:future}

\subsection{WiFi and Mobile Application Integration}

The current system uploads data to ThingSpeak every 15~seconds via HTTP GET. A natural extension is to add a mobile application (Android or iOS) that subscribes to MQTT topics published by the ESP32 and delivers push notifications to healthcare workers when a CRITICAL alert fires. Integration with platforms such as Firebase or AWS IoT Core would support multi-device fleet dashboards accessible from any browser.

\subsection{Hospital and Clinical Deployment}

In a hospital setting, each vaccine transport trolley could carry an ESP32 node monitored from a central nurse station dashboard over the hospital Wi-Fi network. Audit trails persisted in a cloud database would satisfy cold-chain regulatory compliance requirements (WHO PQS, GDP), replacing manual temperature log books and reducing the risk of human error.

\subsection{Machine Learning and Predictive Analytics}

Historical telemetry from multiple deployed nodes—temperature trajectories, shock event counts, and damage score time-series—could train machine learning models to predict thermal failure or container damage before CRITICAL thresholds are reached. Edge-based anomaly detection using TensorFlow Lite on the ESP32 could flag statistically abnormal drift patterns, enabling pre-emptive rerouting of shipments.

\subsection{Cold Chain Pharmaceutical Monitoring}

The same hardware node can be adapted for other temperature-sensitive pharmaceuticals beyond vaccines: insulin ($2$--$8\,^\circ$C), blood products ($1$--$6\,^\circ$C), and monoclonal antibodies ($-80\,^\circ$C with dry ice). Threshold parameters are firmware-configurable and require no hardware changes, making the platform reusable across pharmaceutical cold-chain verticals.

\subsection{Large-Scale Deployment and Scalability}

At national scale, a mesh of ESP32 nodes in vaccine transport vehicles can report to a central dashboard, enabling real-time visibility over entire immunization supply chains. Each node costs approximately INR~1,200--1,500; deploying 1000 units across a state distribution network would cost approximately INR~12--15~lakh, a fraction of the cost of commercial cold-chain management systems. GPS integration would further enable geographic correlation of damage events with road quality or driver behaviour analytics.

% ─── 8. Conclusion ──────────────────────────────────────────────────────────
\section{Conclusion}
\label{sec:conclusion}

This paper presented the design, implementation, and simulation-based validation of a Smart Vaccine Delivery System using an ESP32 microcontroller. The system integrates five sensor modalities—temperature, humidity, acceleration, gyroscopic speed, gas level, and tamper detection—within a single IoT node. The novel Damage Score Algorithm aggregates these modalities into a composite risk metric, enabling both fine-grained per-fault status classification and CRITICAL-level multi-fault escalation. Cloud logging via ThingSpeak provides remote visibility and historical analytics, while a real-time web dashboard enables immediate operational response.

All seven simulated scenarios were successfully validated, demonstrating 100\% detection accuracy in simulation. The system is estimated to cost approximately INR~1,200--1,500 per unit, making it practical for deployment across vaccine cold chains in developing regions.

\smallskip
\noindent\textbf{Future Work:} Planned extensions include: (i) deployment and field validation with real MPU6050 and MQ-135 sensors; (ii) integration with a blockchain ledger for tamper-proof chain-of-custody logging; (iii) GPS-based route safety analytics correlating shock events with geographic locations; (iv) mobile push notifications via a Telegram bot or ThingsBoard rule engine; and (v) machine learning-based predictive thermal failure modeling at the edge.
